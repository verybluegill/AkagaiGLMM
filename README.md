# AkagaiGLMM

## 概要

このリポジトリは、アカガイ操業日誌データを用いて、CPUE（単位努力量当たり漁獲量）の標準化と年変動の可視化を行うための解析ワークフローをまとめたものです。

本リポジトリでは、操業日誌データに対して以下を段階的に実施します。

- 生データのチェック
- 解析用データのクリーニングと再構造化
- GLMM（一般化線形混合モデル）によるCPUE標準化
- 候補モデルの比較
- 解釈・報告用の表および図の出力

同じ構造のデータが与えられれば、再現的に解析できるように設計しています。

---

## 解析フロー

スクリプトは、以下の順に実行します。

`01_check_raw_data.R` → `02_make_clean_data_and_figures.R` → `03_fit_glmm_models.R`

### 各ステップの内容

#### `R/01_check_raw_data.R`
`ActualData/Akagai_sheet.xlsx` を読み込み、生データの値を点検します。  
日付、漁区、操業時間、速度、深度、漁獲数などのチェックを行い、確認用の表と図を `output/check_tables/` および `output/check_figures/` に保存します。

#### `R/02_make_clean_data_and_figures.R`
チェック結果をもとに、解析用データを整形します。  
GLMM入力用データセットを作成するとともに、年別CPUE、サイズ別CPUE、漁区別CPUEなどの集計表・図を出力します。

#### `R/03_fit_glmm_models.R`
`data_processed/akagai_glmm_input.csv` を読み込み、候補GLMMを当てはめます。  
AIC により候補モデルを比較し、選択されたモデルに基づいて標準化CPUE指数、raw relative CPUE との比較図、診断結果などを出力します。

## 結果の見方

本リポジトリでは、各サイズ区分について以下の 2 つを比較しています。

- **Raw relative CPUE**  
  観測された漁獲量と操業量から直接計算した、未補正の相対CPUE

- **Standardized index**  
  GLMM を用いて、月・漁区・船・深度などの影響をできるだけ調整した後の年指数

raw relative CPUE は直感的でわかりやすい一方、操業場所や時期、船ごとの違いの影響もそのまま含みます。  
一方、standardized index は、それらの影響をある程度切り分けたうえで、**年ごとの相対的な変化を見やすくした指標**です。

そのため、この README では、**「見かけの増減」と「条件をそろえて見た増減」がどの程度一致しているか**を見ることを重視しています。

---

## GLMM を使う理由

この解析では、CPUE の標準化に **GLMM（一般化線形混合モデル）** を用いています。

GLMM の主な利点は次のとおりです。

- **操業条件の違いを調整できる**  
  同じ年でも、操業した月、漁区、船、深度が異なれば、単純なCPUE比較は公平ではありません。GLMM を使うことで、こうした条件差の影響を調整しやすくなります。

- **船や漁区ごとのばらつきを扱える**  
  漁区差や船差は、毎年まったく同じとは限りません。GLMM では、こうしたばらつきをランダム効果として扱うことができ、年変動の推定を安定させやすくなります。

- **CPUE のような計数データに対応しやすい**  
  漁獲個体数は 0 を含み、ばらつきが大きくなりやすいデータです。本解析では負の二項分布を用いており、単純な正規分布モデルより実データに合いやすい設定です。

- **年の効果を取り出しやすい**  
  最終的に知りたいのは、操業条件の違いをできるだけ取り除いたうえでの年ごとの相対的な変化です。GLMM はそのための実務的な方法として有用です。

要するに、GLMM を使う理由は、**操業条件の違いに引っ張られすぎずに、年ごとの変化を比較するため**です。

## モデル比較の概要

各応答変数について、年・月・深度・漁区・船の扱いを変えた候補モデルを比較し、AIC に基づいて最良モデルを選択しました。

公開している `best_model_summary.csv` では、次のモデルが選択されています。

- `chu`：`year + month + depth_cat + offset + (1|area) + (1|vessel)`
- `dai`：`year + month + depth_cat + offset + (1|area) + (1|vessel)`
- `toku`：`year + month + depth_quad + offset + (1|area) + (1|vessel)`
- `tokudai`：`year + month + offset + (1|area) + (1|vessel)`
- `count_total`：`year + month + offset + (1|area) + (1|vessel)`

この結果は、**深度の影響の出方がサイズによって異なる可能性**を示しています。  
ただし、これは今回のデータと候補モデルの範囲での結果であり、深度が生物学的に無関係だと断定するものではありません。

## 結果の要約

下図は、4つのサイズ区分（`chu`, `dai`, `toku`, `tokudai`）について、未補正の raw relative CPUE と、GLMM による standardized index を比較したものです。

![Raw relative CPUE and standardized index](docs/results/overlay_best_combined_2x2.png)

全体として、raw relative CPUE と standardized index の年変動パターンは大きくは矛盾しておらず、**観測された増減傾向の多くは、操業条件を調整した後も概ね残る**ことが示されました。  
一方で、一部の年やサイズでは両者に差があり、**単純集計だけでは見えにくい操業条件の偏り**が含まれていることもわかります。

## 結果の考察

### 1. 2023年は多くのサイズで高い水準を示した

`chu`、`dai`、`toku` では、2023年に raw relative CPUE と standardized index の両方が高くなっています。  
このことから、2023年の高水準は、単に有利な月・場所・船で多く操業した結果だけではなく、**条件をそろえて見ても相対的に高かった可能性**があります。

この図からまず読み取りやすいのは、**2023年は中〜大きめサイズで比較的良い年だった可能性がある**という点です。

### 2. tokudai は 2023年から 2024年にかけて高水準が続いた

`tokudai` では、2020年から2022年は低めですが、2023年に上がり、2024年も高い値を示しています。  
raw relative CPUE と standardized index が大きくは離れていないため、**大型個体の増加傾向は比較的頑健なシグナルである可能性**があります。

ただし、これをそのまま「資源全体が回復した」と読むことはできません。大型個体の増加は、成長の進み方、加入の強弱、漁場利用の変化、漁獲対象サイズの偏りなど、複数の要因で見え方が変わります。  
ここでは、**近年は特大サイズの相対的な出現が強まっている可能性がある**と解釈するのが適切です。

### 3. chu・dai では raw CPUE と標準化指数の差がやや見られる年がある

`chu` と `dai` では、2020年や2021年などで raw relative CPUE と standardized index にずれがあります。  
これは、**操業月、操業漁区、船差、深度差などが未補正CPUEに影響していた**ことを示唆します。

つまり、単純な年平均だけを見ると「減った」「増えた」と見える場合でも、条件を調整すると見え方が少し変わることがあります。  
この点が、GLMM による標準化の大きな意義です。

### 4. 深度の重要性はサイズによって異なる

モデル比較では、`chu` と `dai` では深度カテゴリを含むモデル、`toku` では深度の二次効果を含むモデルが選ばれました。  
一方、`tokudai` と `count_total` では深度を含まないモデルが選ばれました。

この結果は、**小型〜中型サイズでは深度の影響が比較的強く現れやすく、大型サイズや全体合計では他の要因のほうが重要だった可能性**を示しています。  
ただし、これは「大型では深度が効かない」と断定するものではなく、今回のデータ範囲では**深度を入れてもモデル改善が小さかった**という意味です。

### 5. この結果は「相対変化」の把握に向いている

ここで示している標準化指数は、資源量そのものを直接推定したものではなく、**各年の相対的な高低を比較するための指標**です。  
したがって、この README での解釈は、

- どの年が相対的に高かったか
- どのサイズで増減パターンが違うか
- 単純集計と標準化結果がどの程度一致するか

を見ることに重点があります。

## 解析上の注意点

### 深度欠損の扱い

深度が欠損または利用不能なレコードは、前処理段階で `NA` として保持されます。  
そのため、深度を含むモデルの比較や当てはめは、**深度情報が利用可能な行**に基づいて実施されます。

### speed / duration の置換ルール

`R/02_make_clean_data_and_figures.R` では、以下の実務的な置換ルールを用いています。

- `speed` は、欠損、`<= 0.5`、または `> 5` の場合に `3` ノットへ置換
- `duration` は、欠損、`<= 10` 分、または `> 90` 分の場合に `60` 分へ置換

これらは解析入力を安定化するためのルールであり、結果の解釈ではこの前処理を前提とする必要があります。

### モデル比較は同一サブセット上で実施

候補モデル間の AIC 比較は、各応答変数ごとに**同一のデータサブセット**上で行っています。  
これは、AIC の差がデータの違いではなく、モデル構造の違いを反映するようにするためです。
